# Task: Fix DH Key Exchange Between Bridge and Server

## Problem

The bridge and server cannot complete a Diffie-Hellman key exchange. When encryption is enabled (`config.json: encryption_enabled: true`), the bridge sends CRYPTOINIT but the handshake fails. The server starts encrypting all responses (including the DHREPLY itself), which the bridge can't decrypt. Result: bridge logs "Received encrypted data but no key" for every message.

## Root Cause — Two Bugs

### Bug 1: Server uses wrong DH parameters (server_message_handler.py:1921)

The server's `_handle_cryptoinit()` creates a NEW `DHExchange()` with fresh random p and g, ignoring the client's p and g from the CRYPTOINIT message. Both sides must use the SAME p and g for the shared secret to match.

```python
# BROKEN (line 1921): new p, g — shared key won't match client's
dh = DHExchange()

# FIX: use client's p and g
dh = DHExchange(p=p, g=g)  # or however DHExchange accepts custom params
```

Check `packages/csc-shared/crypto.py` DHExchange constructor — it may need a way to accept p and g as arguments. If it doesn't support that, add it.

### Bug 2: Server encrypts the DHREPLY (server_message_handler.py:1927-1934)

The server stores `encryption_keys[addr]` on line 1927 BEFORE sending the DHREPLY on line 1934. The `sock_send` override in `server.py` auto-encrypts any outbound data when a key exists for that address. So the DHREPLY gets encrypted — but the client can't decrypt it because it hasn't completed the handshake yet.

```python
# BROKEN order:
self.server.encryption_keys[addr] = shared_key  # key stored
reply = dh.format_reply_message()
self.server.sock_send(reply.encode("utf-8"), addr)  # encrypted! client can't read it

# FIX: send reply BEFORE storing the key
reply = dh.format_reply_message()
self.server.sock_send(reply.encode("utf-8"), addr)  # plaintext — client can parse it
self.server.encryption_keys[addr] = shared_key  # NOW enable encryption
```
